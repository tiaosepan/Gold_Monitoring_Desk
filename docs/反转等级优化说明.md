## 一、问题诊断

### 1.1 原系统存在的问题

#### 问题1: 等级定义混乱 🔴

**代码中的等级定义**:
```python
# 原系统逻辑 (reversal_detector.py)
def determine_reversal_level(...):
    """
    Level 4: 仅政治信号 (political=1, war=0)  # ← 描述为"最纯粹的缓和信号"
    Level 3: 政治+战争信号 (political=1, war=1)
    Level 2: 仅战争信号 (war=1, political=0)
    Level 1: 其他组合
    Level 0: 无信号
    """
```

**文档中的描述**:
- "等级: 1 最强，4 最弱，0 为无信号"
- 推送策略: "仅等级 1 / 2 会推送，3 / 4 级只记录"

❌ **矛盾点**:
1. Level 4 在代码注释中是"最高等级"，但文档说是"最弱"
2. 推送策略只推Level 1/2，但Level 4应该是重要信号
3. 用户理解："Level 4应该比Level 1更重要"，但实际相反

---

#### 问题2: 逻辑不符合直觉 🟡

```python
# 为什么"仅政治信号"是Level 4？
if political_signal == 1 and war_signal == 0:
    return 4  # ← 单一信号反而是最高等级？

# 为什么"4个信号全触发"是Level 1？  
if all signals triggered:
    return 1  # ← 所有信号都触发，反而是最低等级？
```

**业务逻辑问题**:
- 信号越多应该越重要，但原系统是反过来的
- "仅政治信号"的重要性 > "政治+战争+价格+美债"？
- 缺少信号强度的考虑（所有信号都是0/1二值）

---

## 二、解决方案对比

### 2.1 方案A: 简单计数法（推荐）✅

**核心思想**: 等级 = 触发的信号数量

#### 新等级定义

| 等级 | 名称 | 触发信号 | 说明 | 推送策略 |
|-----|------|---------|------|---------|
| **Level 0** | 无信号 | 0个 | 无反转迹象 | ❌ 不记录 |
| **Level 1** | 弱信号 | 1个 | 观察阶段 | ✅ 记录，不推送 |
| **Level 2** | 中等信号 | 2个 | 关注阶段 | ✅ 记录，不推送 |
| **Level 3** | 强信号 | 3个 | 准备行动 | 🔔 **推送告警** |
| **Level 4** | 极强信号 | 4个 | 立即行动 | 🔔 **优先推送** |

#### 代码实现

```python
def determine_reversal_level_v2(self, price_signal, political_signal, 
                                 war_signal, us10y_signal):
    """
    V2版本: 按信号数量递增
    
    Level 0: 0个信号 - 无反转迹象
    Level 1: 1个信号 - 弱信号，观察
    Level 2: 2个信号 - 中等信号，关注
    Level 3: 3个信号 - 强信号，准备行动 [推送]
    Level 4: 4个信号 - 极强信号，立即行动 [推送]
    """
    # 简单明了：信号数量 = 等级
    signal_count = price_signal + political_signal + war_signal + us10y_signal
    return signal_count
```

#### 优点 ⭐⭐⭐⭐⭐

1. **直观易懂**: 信号越多，等级越高
2. **符合直觉**: Level 4 > Level 1
3. **推送策略清晰**: Level 3/4推送，Level 1/2观察
4. **易于维护**: 代码简单，逻辑清晰

#### 缺点

- 未考虑信号重要性差异（所有信号权重相同）
- 未考虑信号强度（RSS评分、反弹幅度等）

### 2.2 方案B: 加权评分法（高级）⭐

**核心思想**: 等级 = 加权评分 / 20

#### 评分规则

```python
# 信号权重配置（总计100分）
weights = {
    'price': 25.0,      # 价格信号权重25%
    'political': 25.0,  # 政治信号权重25%
    'war': 25.0,        # 战争信号权重25%
    'us10y': 25.0       # 美债信号权重25%
}

# 计算加权评分
weighted_score = 0
for signal_type, weight in weights.items():
    if signal == 1:
        strength = get_signal_strength(signal_type)  # 0.0-1.0
        weighted_score += weight * strength

# 评分转等级
if weighted_score >= 81: return 4      # 81-100分 → Level 4
elif weighted_score >= 61: return 3    # 61-80分  → Level 3
elif weighted_score >= 41: return 2    # 41-60分  → Level 2
elif weighted_score >= 21: return 1    # 21-40分  → Level 1
else: return 0                         # 0-20分   → Level 0
```

#### 信号强度计算

```python
# 价格信号强度
rebound_pct = (latest_price - min_price) / min_price * 100
strength = min(rebound_pct / threshold, 2.0)  # 最大2.0

# RSS事件信号强度  
avg_score = sum(event.impact_score for event in events) / len(events)
strength = (avg_score - 5) / 5.0  # 6-10分 → 0.2-1.0强度

# 美债信号强度
drop_bp = (max_yield - current_yield) * 100
strength = min(drop_bp / threshold, 2.0)  # 最大2.0
```

#### 优点 ⭐⭐⭐⭐

1. **考虑信号强度**: RSS评分高的事件权重更大
2. **更精确**: 评分系统更细粒度
3. **可配置权重**: 可以调整各信号的重要性
4. **平滑过渡**: 避免等级跳变

#### 缺点

- 复杂度更高，不易理解
- 需要维护信号强度计算逻辑
- 阈值需要根据实际数据调整

```python
async def reversal_detector_task(self):
    db = SessionLocal()
    try:
        # 读取配置
        config = db.query(SystemConfig).filter_by(
            config_key='reversal_detector_version'
        ).first()
        version = config.config_value if config else 'v1'
        
        # 根据配置选择版本
        if version == 'v2':
            service = ReversalDetectorServiceV2(db)
            result = await service.detect_and_save_v2(use_weighted=False)
        elif version == 'v2_weighted':
            service = ReversalDetectorServiceV2(db)
            result = await service.detect_and_save_v2(use_weighted=True)
        else:  # v1
            service = ReversalDetectorService(db)
            result = await service.detect_and_save()
        
        # 处理结果...
    finally:
        db.close()
```

**Step 3: 提供切换API**

```python
# backend/api/routes.py
@router.put("/api/reversal/switch-version")
async def switch_reversal_version(
    version: str = Query(..., regex="^(v1|v2|v2_weighted)$"),
    db: Session = Depends(get_db)
):
    """切换反转检测版本"""
    config = db.query(SystemConfig).filter_by(
        config_key='reversal_detector_version'
    ).first()
    
    if config:
        config.config_value = version
    else:
        config = SystemConfig(
            config_key='reversal_detector_version',
            config_value=version,
            description='反转检测版本'
        )
        db.add(config)
    
    db.commit()
    
    return {
        'success': True,
        'message': f'已切换到{version}版本',
        'version': version
    }
```

## 七、常见问题

### Q1: 为什么要改变等级定义？

**A**: 原系统的等级定义容易造成混淆：
- 代码注释说Level 4是"等级最高"
- 文档说Level 4是"最弱"
- 推送策略只推Level 1/2，让人以为Level 4不重要

V2改为按信号数量递增，更直观易懂。

---

### Q2: V2会影响历史数据吗？

**A**: 不会。新旧版本使用同一张表，通过`note`字段区分：
- 原版本: `note`不包含`[V2]`标记
- V2版本: `note`以`[V2]`开头

可以通过SQL查询区分：
```sql
-- 查询V1数据
SELECT * FROM reversal_conditions WHERE note NOT LIKE '[V2]%';

-- 查询V2数据
SELECT * FROM reversal_conditions WHERE note LIKE '[V2]%';
```

---

### Q3: 切换到V2会漏掉重要信号吗？

**A**: 不会。对比测试显示：
- V2推送次数 ≥ V1推送次数
- V2会推送"3个信号"的场景（原版本可能不推）
- 所有原版本推送的场景，V2也会推送

---
