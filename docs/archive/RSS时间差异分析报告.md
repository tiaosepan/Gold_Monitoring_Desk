## 问题根源分析

### 核心问题：RSS源内容滚动更新

RSS源的内容会不断更新，旧的条目会被新的条目替换。这导致：

1. **时间线**:
   - 01:15 - RSS源发布了一批事件（复刻系统采集到）
   - 06:29 - RSS源发布了新一批事件（原系统采集到）
   - 07:43 - 原系统进行采集，获取到06:29的事件
   - 08:40 - 复刻系统进行采集，但RSS源已经更新
   - 09:27 - 复刻系统再次采集，获取到最新的事件

2. **RSS源特性**:
   - RSS源通常只保留最近的N条记录（如20-50条）
   - 当新内容发布时，旧内容会被移出RSS feed
   - 这是RSS协议的正常行为

3. **采集频率影响**:
   - 原系统可能采集频率更高，或者恰好在RSS源更新后立即采集
   - 复刻系统的采集频率是30分钟一次（配置在scheduler中）

## 解决方案

### 方案1: 增加采集频率 ✅ **推荐**

**当前配置**: 30分钟一次

**建议配置**: 15-20分钟一次

**修改位置**: `backend/scheduler.py`

```python
# 当前
self.scheduler.add_job(
    self.rss_collector_task,
    trigger=IntervalTrigger(minutes=30),
    ...
)

# 建议
self.scheduler.add_job(
    self.rss_collector_task,
    trigger=IntervalTrigger(minutes=15),  # 改为15分钟
    ...
)
```

**优点**:
- 减少错过RSS更新的概率
- 更及时地获取最新事件
- 与原系统行为更接近

**缺点**:
- 增加服务器负载（但影响很小）
- 可能采集到更多重复事件（已有去重机制）

### 方案2: 保持现状 ✅ **可接受**

**理由**:
1. RSS源的滚动更新是**正常现象**，不是系统缺陷
2. 两个系统采集到不同批次的事件是**预期行为**
3. 只要关键词匹配、评分逻辑一致，事件内容差异是可接受的
4. 随着时间推移，两个系统都会积累足够的事件
