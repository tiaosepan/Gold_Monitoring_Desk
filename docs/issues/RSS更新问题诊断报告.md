## 为什么看起来"没有更新"？

### 原因分析

1. **服务器崩溃期间错过了采集**
   - 10:43那一轮采集时，RSS源中有"基建损毁"事件（发布02:30）
   - 原系统采集到了，本地系统因崩溃错过了

2. **RSS Feed的滚动更新**
   - RSS源不断有新内容发布（03:35, 03:32等）
   - 旧内容被挤出feed（"基建损毁"事件已不在feed中）
   - 新内容中很多**不命中关键词**，所以不会被保存

3. **关键词过滤机制**
   - 本地系统严格按关键词过滤（完全匹配原系统逻辑）
   - 只有命中政治/战争关键词的事件才会被保存
   - 例如："离岸人民币拆息"、"花旗投资者乐观" 这类金融新闻不会被采集

## 根本问题：API排序错误

### 问题发现

经过深度诊断发现：
- ✅ **数据库中有最新事件**（ID:75-82，采集时间11:38）
- ❌ **API返回的却是旧事件**（ID:51，采集时间09:27）

**原因**: API使用了错误的排序字段！

### 错误的排序逻辑

**文件**: `backend/api/routes.py` (第662-664行)

```python
# 错误的排序（只按published_at）
events = query.order_by(
    RSSEvent.published_at.desc()
).limit(limit).all()
```

**问题**:
- 按 `published_at`（RSS发布时间）排序
- 但有些事件的发布时间可能比采集时间早很多
- 导致新采集的事件反而排在后面

### 正确的排序逻辑

**修复后**:

```python
# 正确的排序（先按fetched_at，再按published_at）
events = query.order_by(
    RSSEvent.fetched_at.desc(),
    RSSEvent.published_at.desc()
).limit(limit).all()
```

**改进**:
- ✅ 先按 `fetched_at`（采集时间）降序排序
- ✅ 相同采集时间的，再按 `published_at` 降序排序
- ✅ 完全匹配原系统的排序逻辑

## 技术说明

### RSS Feed的特性

RSS feed是一个**滚动窗口**：
- 只保留最新的N条记录（通常20-50条）
- 新内容发布后，旧内容会被挤出
- 如果错过采集时机，旧内容就无法再获取

### 本地系统的过滤逻辑

```python
# 只保存命中关键词的事件
if not matched_keywords:
    continue  # 跳过不命中的事件
```

这个逻辑**完全匹配原系统**，确保事件流只显示与黄金反转相关的政治/战争事件。
