**修复时间**: 2026-03-26 12:30  
**问题**: RSS事件流和RSS抓取记录不显示

---

## 问题描述

用户反馈：
1. ✅ API返回了最新事件（ID:78-82，采集时间11:38）
2. ❌ 前端"RSS事件流"列表中不显示这些事件
3. ❌ "RSS抓取记录"没有数据

---

## 根本原因

### 原因1: API排序错误（影响前端显示）

**问题**: 两个API端点都使用了错误的排序字段

#### 位置1: `/api/reversal/events` (第662-664行)

```python
# 错误：只按published_at排序
events = query.order_by(
    RSSEvent.published_at.desc()
).limit(limit).all()
```

#### 位置2: `/api/reversal/status` (第526-528行)

```python
# 错误：只按published_at排序
recent_events = db.query(RSSEvent).order_by(
    RSSEvent.published_at.desc()
).limit(50).all()
```

**后果**:
- 新采集的事件（fetched_at: 11:38）的发布时间（published_at: 03-25 19:xx）比旧事件（published_at: 03-26 01:xx）要早
- 按 `published_at` 排序时，旧事件反而排在前面
- 前端获取到的是旧事件，看起来"没有更新"

### 原因2: source字段类型错误

**问题**: `/api/reversal/status` 第566行返回了对象而不是字符串

```python
'source': e.source,  # 错误：这是一个SQLAlchemy对象
```

**后果**: 前端无法正确显示来源名称

### 原因3: 缺少运行记录

**问题**: `ReversalDetectorService.detect_and_save()` 没有记录到 `UpdateRecord` 表

**后果**: 
- `recent_runs` 始终为空
- 前端"反转运行记录"无数据显示

---

## 修复方案

### 修复1: 统一API排序逻辑

**文件**: `backend/api/routes.py`

#### 修复 `/api/reversal/events` (第662-665行)

```python
# 修改后：先按fetched_at，再按published_at
events = query.order_by(
    RSSEvent.fetched_at.desc(),
    RSSEvent.published_at.desc()
).limit(limit).all()
```

#### 修复 `/api/reversal/status` (第526-529行)

```python
# 修改后：先按fetched_at，再按published_at
recent_events = db.query(RSSEvent).order_by(
    RSSEvent.fetched_at.desc(),
    RSSEvent.published_at.desc()
).limit(50).all()
```

### 修复2: 修正source字段

**文件**: `backend/api/routes.py` (第566行)

```python
# 修改前
'source': e.source,

# 修改后
'source': e.source.name if e.source else 'Unknown',
```

### 修复3: 添加反转检测运行记录

**文件**: `backend/services/reversal_detector.py`

#### 成功时记录 (第275行后)

```python
self.db.add(reversal_record)
self.db.commit()

# 新增：记录运行日志
from ..database import UpdateRecord
update_record = UpdateRecord(
    data_type='reversal_detection',
    status='success',
    record_count=1
)
self.db.add(update_record)
self.db.commit()
```

#### 失败时记录 (第295行后)

```python
# 新增：记录失败日志
from ..database import UpdateRecord
update_record = UpdateRecord(
    data_type='reversal_detection',
    status='failed',
    record_count=0,
    error_message=str(e)
)
self.db.add(update_record)
self.db.commit()
```

## 技术说明

### 为什么需要按fetched_at排序？

RSS事件的 `published_at` 是RSS源发布时间，可能是几小时甚至几天前。如果只按 `published_at` 排序：

**示例**:
- 事件A: fetched_at=11:38, published_at=03-25 19:00（昨天晚上7点发布）
- 事件B: fetched_at=09:27, published_at=03-26 01:00（今天凌晨1点发布）

按 `published_at` 排序：事件B排在前面（虽然它是旧的采集）  
按 `fetched_at` 排序：事件A排在前面（最新采集的）✅

### 原系统的排序逻辑

通过分析原系统API响应，确认其排序逻辑：

```python
.order_by(
    RSSEvent.fetched_at.desc(),      # 优先按采集时间
    RSSEvent.published_at.desc()     # 相同采集时间再按发布时间
)
```
